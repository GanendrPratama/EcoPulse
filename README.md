# EcoPulse

**EcoPulse** is a NoSQL mini-project application for a **Smart City Environmental Sensor Network**. The system simulates environmental sensors across a city and stores time-series sensor readings in **Apache Cassandra**, a wide-column NoSQL database designed for high write throughput and horizontally scalable workloads.

This project was developed for a NoSQL Mini Project using the following stack:

- **Frontend:** React JS, Tailwind CSS, Recharts
- **Backend:** Express JS, Node.js
- **Database:** Apache Cassandra
- **Infrastructure:** Docker Compose, Nginx
- **Data Source:** Synthetic IoT sensor data generated by a local simulator

## Project Overview

Modern cities can use environmental sensors to monitor air quality, temperature, humidity, PM2.5 concentration, and sensor battery status in real time. However, when thousands of sensors send data every few seconds, a traditional relational database can become a bottleneck because the workload is highly **write-heavy** and grows continuously over time.

EcoPulse addresses this problem by using **Apache Cassandra**, a wide-column database that is suitable for distributed, high-volume, time-series workloads. The application demonstrates how Cassandra can be used to store and retrieve environmental sensor readings efficiently based on sensor ID and timestamp.

---

## Real-World Case Study

The real-world case study used as inspiration for this project is **Discord's message storage system**.

Discord processes a very large number of messages and needs a storage system that supports:

- High write throughput
- Low-latency reads
- Horizontal scalability
- Data retrieval based on query patterns
- High availability across distributed infrastructure

This is conceptually related to EcoPulse because both systems handle large volumes of continuously generated records:

| Case Study | Data Type | Main Access Pattern |
|---|---|---|
| Discord | Chat messages | Fetch recent messages by channel |
| EcoPulse | Sensor readings | Fetch recent readings by sensor |

In both cases, Cassandra's wide-column model is suitable because data can be partitioned by an entity identifier and sorted by time.

## Database Design

EcoPulse uses Cassandra with a **query-first data modeling approach**. The main query requirement is:

> Retrieve recent or historical readings for a specific sensor efficiently.

### Keyspace

```sql
CREATE KEYSPACE IF NOT EXISTS ecopulse
WITH replication = {
  'class': 'SimpleStrategy',
  'replication_factor': 1
};
```

For local development, the project uses `SimpleStrategy` with replication factor `1`. In a real distributed production environment, `NetworkTopologyStrategy` would be more appropriate.

### Table: `sensor_readings`

```sql
CREATE TABLE IF NOT EXISTS sensor_readings (
  sensor_id TEXT,
  recorded_at TIMESTAMP,
  temperature_c FLOAT,
  humidity_percent INT,
  aqi_value INT,
  pm25_level FLOAT,
  battery_status INT,
  PRIMARY KEY ((sensor_id), recorded_at)
) WITH CLUSTERING ORDER BY (recorded_at DESC);
```

### Primary Key Explanation

| Key Part | Column | Purpose |
|---|---|---|
| Partition Key | `sensor_id` | Groups all readings from the same sensor together |
| Clustering Key | `recorded_at` | Sorts readings by timestamp within each sensor partition |

This model fits Cassandra because the application commonly queries data by `sensor_id` and retrieves the latest readings in timestamp order.

## How to Run the Project

### Prerequisites

Make sure the following tools are installed:

- Docker Desktop
- Docker Compose
- Git

### 1. Clone the Repository

```bash
git clone <repository-url>
cd EcoPulse
```

### 2. Build and Run All Services

```bash
docker compose up --build -d
```

This command starts:

- Cassandra database
- Cassandra schema initializer
- Express backend API
- Sensor simulator
- React frontend served by Nginx

### 3. View Logs

```bash
docker compose logs -f
```

To view logs from a specific service:

```bash
docker compose logs -f cassandra-init
docker compose logs -f backend
docker compose logs -f simulator
docker compose logs -f frontend
```

### 4. Open the Application

Frontend dashboard:

```text
http://localhost:8080
```

Backend health check:

```text
http://localhost:3000/api/health
```

Expected health response:

```json
{
  "status": "healthy",
  "cassandra": "connected"
}
```

### 5. Stop the Application

```bash
docker compose down
```

To stop the application and remove Cassandra volume data:

```bash
docker compose down -v
```

## API Endpoints

### Health Check

```http
GET /api/health
```

Checks whether the backend can connect to Cassandra.

### Insert Sensor Data

```http
POST /api/sensors/data
```

Example request body:

```json
{
  "sensor_id": "JKT-AQI-001",
  "temperature_c": 29.5,
  "humidity_percent": 70,
  "aqi_value": 120,
  "pm25_level": 45.2,
  "battery_status": 88
}
```

### Get Latest Readings

```http
GET /api/sensors/latest
```

Returns the latest reading from each known sensor.

### Get Sensor History

```http
GET /api/sensors/:sensor_id/history?limit=50
```

Example:

```http
GET /api/sensors/JKT-AQI-001/history?limit=50
```

Optional query parameters:

| Parameter | Description |
|---|---|
| `limit` | Maximum number of readings to return |
| `start_time` | Filter readings after a given timestamp |
| `end_time` | Filter readings before a given timestamp |

## Benchmark: Cassandra vs PostgreSQL

This section presents the results of a write and read performance benchmark comparing **Apache Cassandra** (used in EcoPulse) against **PostgreSQL** (a traditional relational database) under the same workload conditions.

The goal of this benchmark is to validate the technology choice of using Cassandra for a high write-throughput, time-series sensor data workload.

---

### Benchmark Configuration

Both databases were tested using the same parameters.

#### Write Benchmark Configuration

| Parameter | Value |
|---|---:|
| total_records | 10,000 |
| concurrency | 100 |
| sensor_count | 100 |
| warmup_records | 200 |

#### Read Benchmark Configuration

| Parameter | Value |
|---|---:|
| total_queries | 2,000 |
| concurrency | 100 |
| sensor_count | 100 |
| history_limit | 50 |

#### PostgreSQL Configuration

| Parameter | Value |
|---|---|
| version | PostgreSQL 16.14 |
| host | postgres |
| port | 5432 |
| db_name | ecopulse |
| pool_max | 100 |
| synchronous_commit | on |

#### Cassandra Configuration

| Parameter | Value |
|---|---|
| host | cassandra |
| datacenter | dc1 |
| keyspace | ecopulse |
| consistency | LOCAL_ONE |

---

### Write Performance Results & Write Latency Distribution (ms)

![image](https://hackmd.io/_uploads/r1xiSJhkMe.png)


**Analysis:** Cassandra completed 10,000 writes with zero errors in 852ms, achieving a throughput of ~11,730 writes/sec — roughly **2.5× faster** than PostgreSQL. PostgreSQL also had 33 errors (0.33% error rate) and a significantly higher p99 and max latency, indicating occasional slowdowns under concurrent write load.

---

### Read Performance Results & Read Latency Distribution (ms)

![image](https://hackmd.io/_uploads/H1a1UJhJMg.png)

**Analysis:** Cassandra completed 2,000 reads with zero errors, achieving ~2,934 reads/sec — roughly **2.1× faster** than PostgreSQL. PostgreSQL had 10 read errors and significantly higher tail latencies (p99: 782ms vs 191ms), which in a production environment would directly impact dashboard responsiveness.

---

### Summary

| Aspect | PostgreSQL | Cassandra | Winner |
|---|---|---|---|
| Write throughput | 4,627 writes/sec | 11,730 writes/sec |  Cassandra |
| Write error rate | 0.33% | 0% | Cassandra |
| Read throughput | 1,395 reads/sec | 2,934 reads/sec | Cassandra |
| Read error rate | 0.5% | 0% | Cassandra |
| Write p99 latency | 63.8 ms | 26.7 ms | Cassandra |
| Read p99 latency | 782.8 ms | 191.2 ms | Cassandra |

These results confirm that Apache Cassandra is the more suitable database for the EcoPulse use case, particularly for high-volume, concurrent IoT sensor writes and low-latency time-series reads, consistent with the design goals described in the Database Design section.

> **Note:** This benchmark was conducted in a local Docker environment with a single-node Cassandra cluster (`replication_factor: 1`). In a real distributed multi-node deployment, Cassandra's performance advantage over relational databases is expected to be even more pronounced.
